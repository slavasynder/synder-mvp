The Problem

Where TxnRecon Lands Today — and the Gap

Transaction Reconciliation already produces a clear, auditable result. But part of the result is "we couldn't tie these rows" — and today the user has no in-Synder way to act on it.

Four result buckets — recap
✓ Matched
Integration ↔ accounting tied cleanly by ID. Auto-matched.
⚠ Discrepancy
Matched by ID, amounts disagree. User fixes in books → refresh.
↗ Missing in Accounting
Integration sees it. Books don't. No in-Synder action today.
↙ Missing in Integration
Books have an entry. Integration doesn't. No in-Synder action today.
Why rows end up in "Missing" — and why they often do belong together
📝
Different conventions
Bookkeeper recorded the entry with a different label, date, or grouping than the integration sent — no shared ID for auto-matching to tie them.
🪛
Manual entries in QBO
Bookkeeper made the journal entry by hand. The amounts align with the integration's payout, but ID-based matching can't see that.
🔁
Re-syncs & deletions
A row was re-synced, deleted, or rebuilt — leaving an orphan on one side and a near-twin on the other.
Today's workaround. Users either (1) leave Synder, post the missing entry in QBO/Xero, and refresh; or (2) click Ignore on both sides as a stand-in for matching. The first is friction; the second breaks the meaning of Ignore — and the reconciliation looks "finished" only because nothing's actually resolved. We've seen this pattern in 5 LogRocket sessions as the original signal for the work.
The Resolution

Manual Matching — DIS-330, Iteration 1

User-initiated Match / Unmatch on the Missing source surface. Strict eligibility — the bet is "the user can never record a wrong match."

🔗
What ships
  • Multi-select rows on both Missing source tables (uncapped at selection).
  • Match action — paired into a single match record.
  • Unmatch action — both per-row on Manual rows and on every Discrepancy row.
  • Cardinality: 1:1, 1:N, N:1, N:M — all supported.
  • Manual matches are Synder-internal — no write back to QBO/Xero.
🏷
Surfaces — no new tabs
  • Existing Matched tab gets a Match type column.
  • Values: Auto (current system matches), Manual (DIS-330) — plus AI and Journal Entry reserved.
  • Missing source — Match action lives above both tables in a separate action block.
  • Discrepancy — gains Unmatch as a peer to the existing Ignore.
The four "this is what we promise" anchors
✋
User-initiated
Synder never invents a match. The user always confirms.
🚫
No wrong matches
Strict currency + balanced-sum checks block bad confirms.
↩
Reversible
Unmatch returns the rows to Missing source on the same reconciliation.
📜
Auditable
Every match carries its origin label — Auto, Manual, AI*, Journal Entry*.
The Resolution

How It Works — The User Flow

Four steps. The user is in Missing source the whole time.

Step 1
Select rows
Tick checkboxes on the Missing in accounting table and the Missing in integration table. Counters show "N selected" above each.
→
Step 2
Match button
Always visible above both tables. Enabled when both sides have selection. Disabled (with tooltip) in every other state.
→
Step 3
Validate
Same currency. Balanced signed sums (1:1, 1:N, N:1, N:M). ≤200 rows total. Fails → blocked with a clear reason.
→
Step 4
Matched tab
Rows leave Missing source. New record renders on Matched with Match type = Manual.
Match button placement (revised). Originally moved into the Missing source table header on the post-mocks pass — moved back to a separate action block above both tables after design / dev discussion (final state).
Same-currency, balanced sums. Sum-zero pairs (charge + refund on both sides) are allowed — 0 == 0 satisfies the rule. A real cancelled-pair grouping is a legitimate audit-worthy match.
The Resolution

Eligibility — What Blocks a Match

Strict, predictable, deterministic. The whole point of v1 is that the user cannot record a wrong match.

Four eligibility rules
💱
Same currency, across both sides
If a selection spans two distinct currencies (across or within either side), Match is blocked at confirm. Generic error — no row-level surfacing in v1.
⚖
Balanced signed sums
sum(side A) == sum(side B) with raw signed amounts. Blocks with the deltas spelled out so the user can fix the selection.
🚪
No already-matched rows
A row already in any match state (Auto, Manual, future AI/JE) cannot be re-eligible. Unmatch first.
🧮
≤ 200 rows total per action
Cap is action-bound, not selection-bound (checkboxes are uncapped). Over-cap Match attempts → blocking error toast.
200-row cap — where it applies
Manual Match — capped at 200
Toast fires on Match click if total > 200. TxnReconciliationSelectionLimitHit Mixpanel event fires once per toast.
Ignore — uncapped
Ignore is intentionally not capped. Different action, different volume profile.
Future JE creation — will share the cap
DIS-256 (in-Synder JE from Missing source) will adopt the same 200-row per-action cap when it ships, for consistency across manual actions.
The Resolution

Unmatch + Lifecycle

How matches come apart, and what happens on Refresh.

Where Unmatch is surfaced
📋
Matched tab — Manual rows only
Per-row Unmatch on every Manual record. Selection checkboxes + bulk Unmatch via header button when multiple Manual rows are selected.
⚠
Discrepancy tab — every row
Unmatch added alongside the existing Ignore. Breaking an Auto match happens here — both rows return to Missing source.
🚫
Auto rows — no Unmatch, no checkbox
On the Matched tab, Auto rows have no checkbox at all (not a disabled placeholder) and no Unmatch action. Use the Discrepancy tab to break Auto matches.
Refresh + Delete — manual matches don't survive
🔄
Refresh + Delete clear all manual matches
Dev-confirmed v1 constraint: preserving manual matches across a rebuild is not feasible (re-pull from source data is destructive). Cleared rows return to Missing source.
⚠
Warning modal — on every Refresh / Delete click
Generic copy ("all manual changes will be lost"). Always shown, no conditional pre-check for whether matches exist (dev preference — keeps the path simple).
What this means in practice. Users should treat manual matches as "reconciliation-scoped" — they live on the current run, not across refreshes. If we see in adoption data that users redo the same matches every refresh, that's a real signal — and the path to preserving matches becomes a candidate for Iteration 2.
The Resolution

Supporting Changes Across the Page

Filters, column, export, Mixpanel — small additions that the whole TxnRecon page inherits.

🏷
Matched tab — Match type column
New column on the existing Matched tab. Filterable, not sortable in v1. Values:
Auto Manual AI (reserved) Journal Entry (reserved)
📤
Matched export — gains Match type column
No new export options. The existing Matched export adds Match type with the same values as the on-screen column. Other tab exports unchanged.
🔎
Filters on every table on every tab
Currency selector (from a prefilled global list — popular currencies at the top) + amount range (From / To). Applied independently per tab. Currency list does not recompute from current table content — that's intentional (dynamic tables would force constant recalculation).
📡
Mixpanel — seven new events
Attribution: ManuallyMatched, Ignored. Friction: SelectionLimitHit. Journey: RunAuditClicked, StartMatchingClicked, ListViewed, Opened. All prefixed TxnReconciliation*.
One thing not changing: LogRocket. Stays the qualitative-observation tool (blocked-attempt sessions, UX behaviour around the new affordances). Mixpanel handles attribution.

Roadmap

Future Goals for Transaction Reconciliation

DIS-330 is one step. Six prototype directions are queued — the Burkland webinar gave us an early ranking signal on which to ship sooner.

The six prototypes — Burkland's early ranking (9 of 19 attendees voted live)
🥇
Balance Reconciliation
~95%
Aggregate-first lens — verify totals tie before drilling into rows. The depth ask from accountants. DIS-259.
🥈
JE-from-Synder
~80%
Create a JE in-Synder for Missing-in-Accounting rows, no QBO round-trip. Same Missing source surface as DIS-330. DIS-256 / SD-17432.
🥉
Match with AI
~75%
"AI proposes pairs, you decide" — reasoning + confidence score. Landed warmer than we predicted. DIS-370 (separate Opportunity).
4
Configurable Matching Rules
~58%
Deterministic auto-match rules — pair by date tolerance, amount tolerance, payee pattern. DIS-345.
5
Firm-Level Reconciliation
~38%
Cross-client / portfolio view for firms. Burkland ranked it lower — they have their own firm workflows. May matter more to other ICPs.
6
Audit Report
~33%
Formal audit-ready export with provenance per row. Low pull from Burkland — audit is handled elsewhere in their stack.
What this tells us already. The "trust the numbers" reconciliation arc has two clear next stops: aggregate-first verification (Balance Reconciliation) and in-Synder accounting entries (JE-from-Synder). DIS-330 unlocks the surface they'll plug into.
